# Dentist Profile Page Audit (deepest tier / T5‚ÄìT6)

Auditor pass over the individual dentist profile pages at
`/dental/{state}/{market}/{city}/{dentist-slug}/index.html`.
Generated by `seo-build/generate-plans.js` (do NOT hand-edit `dental/`; all fixes go in the generator).

## 1. JSON-LD / Structured data

Strong overall. Single `@graph` with:
- `Dentist` + `LocalBusiness` + `MedicalOrganization` (array `@type`) ‚Äî correct, with
  `@id`, `name`, `url`, `medicalSpecialty`, `PostalAddress`, `telephone`, `GeoCoordinates`,
  `areaServed` (City list), `sameAs`.
- `sameAs` is correctly an ARRAY: `[official website, google maps query URL]`. Good.
- `geo` lat/lon present and matches `geo.position` meta and map links. Good.
- `AggregateRating` with `ratingValue`, `reviewCount`, `bestRating:"5"`.
- `BreadcrumbList` (5 ListItems, positions 1‚Äì5).
- `FAQPage` (6 Qs new template / 3 Qs old template), text mirrors visible accordion. Good (no schema-only FAQ).
- `WebPage` node tying it together with `datePublished`/`dateModified` and `isPartOf #website`.

Findings:
- **AggregateRating provenance is plausibly real** (varied values: 4.4/715, 4.9/126, 4.7/151,
  4.6/310; page copy says "Data sourced from the office's public Google listing"). NOT obviously
  fabricated. BUT: must verify the generator only emits `AggregateRating` when a real
  `google_review_count > 0`. Any page that emits a rating with 0 reviews, or a hardcoded
  default, would violate the no-fabricated-reviews rule and risk a Google structured-data
  manual action. Confirm in generator: omit the whole `aggregateRating` node when
  `weighted_rating`/`google_review_count` are null/0. (Could not find a 0-review sample to
  disprove; flag for code check.) ‚Äî see DEF-4.
- `medicalSpecialty:"Dentistry"` is fine. `priceRange` is absent (minor; LocalBusiness benefits from it).

## 2. Title & meta description

- New template title: `{Name} | PPO Dentist in {City}, {ST} | CoverCapy` and a
  `Top-Rated` variant `{Name} | Top-Rated PPO Dentist in {City}, {ST} | CoverCapy`
  (high-rating offices). Lengths 55‚Äì75 chars ‚Äî **several exceed 60** (e.g.
  "Maldonado Dentistry | Top-Rated PPO Dentist in Treasure Island, FL | CoverCapy" ~78).
- OLD template title: `{Name} | PPO Dentist {City}, {ST} ‚Äî Insurance Verified | CoverCapy`
  ‚Äî **contains an em-dash** (CLAUDE.md forbids em-dashes) and a different formula =>
  inconsistent SERP brand pattern across the corpus.
- Meta descriptions are templated, ~150‚Äì165 chars, name up to "major PPO insurance" + rating +
  "verify free with CoverCapy". Some exceed 155. Pattern is near-identical across every page
  except for {Name}/{City}/{rating} tokens => low differentiation but acceptable for a directory.

## 3. H1 / keyword usage

- H1 = bare practice name only (`<h1>Maldonado Dentistry</h1>`). The "PPO Dentist ¬∑ {City}, {ST}"
  eyebrow carries the keyword, not the H1. The H1 has **no city or "PPO dentist" / "near me"
  modifier**, weakening on-page relevance for "{practice} {city}" and "dentist {city}" queries.
- No "near me" phrasing in H1/H2 (only in an FAQ answer and the "PPO dentists near {City}" rail H2).
  Carrier keywords appear only in JS-injected pills (see DEF-2), not in crawlable headings/body.

## 4. Breadcrumb

- Visible `<nav aria-label="Breadcrumb">` markup matches the `BreadcrumbList` schema exactly:
  CoverCapy ‚Üí {State} ‚Üí {Metro} ‚Üí {City} ‚Üí {Practice} (positions 1‚Äì5). Correct numbering.
- **DEFECT: no "Find a Dentist" / `/dentist` crumb.** Per the audit brief, the national hub crumb
  should sit between Home and State (e.g. CoverCapy ‚Üí Find a Dentist ‚Üí {State} ‚Üí ...). It is absent
  in both schema and visible markup. The `/dentist` hub is also not linked anywhere else on the page,
  so the deepest tier never links up to the national hub (orphaning the hub from 11k inbound links).
- Redundant path segments where metro == city (NY: `/ithaca/ithaca/`, WA: `/seattle/seattle/`).
  Breadcrumb then shows the same word twice; cosmetic but reads oddly and dilutes uniqueness.

## 5. Content depth / thin-content risk ‚Äî HIGH

- The only unique, crawlable prose is the "About this practice" block: ~2 short templated
  sentences (~40‚Äì55 words) = address + "works with major PPO insurance" + generic CoverCapy line.
  Nearly identical across all pages save name/address. **Thin + near-duplicate body content.**
- The substantive differentiators ‚Äî listed carriers, services/treatments, hours, nearby offices ‚Äî
  are **NOT in the static HTML**. They are baked into an inline `var T5 = {...}` JS object and
  injected into empty containers (`#carrier-rows`, `#proc-pills`, `#hours-table`, `#nearby-grid`)
  at runtime. A crawler that does not execute JS sees empty sections. Even rendered, the carrier
  data renders as button rows, not indexable prose.
- Estimated crawlable word count (excluding JS): ~250‚Äì400 words, mostly boilerplate shared by
  thousands of pages. With ~11,396 such pages this is a large doorway/thin-content footprint.

## 6. Internal linking

- Up-links: breadcrumb to State and Metro and City (City via breadcrumb only). Good for those three.
- **No link to the `/dentist` national hub** (orphan risk for the hub).
- **No links to carrier pages** (e.g. /dental-insurance/{carrier}) ‚Äî carriers are JS pills with a
  modal, not anchors. Big missed silo opportunity: 11k pages list carriers but pass no link equity
  to carrier landing pages.
- **No links to per-treatment guides** beyond a fixed generic `/learn/*` and `/dental-insurance/`
  set repeated identically on every page (financing, ppo-vs-cash, network-tiers, verify-ppo, etc.).
- Nearby-offices rail is JS-injected => sibling internal links (the main horizontal-link mechanism)
  are invisible without JS, weakening crawl discovery of neighboring dentist pages.

## 7. Canonical & stale paths

- Canonical is built correctly from parts (state/market/city/slug), trailing slash. No `/dentists/`
  or `seo_path` leakage observed in the sample. Good.
- **DEFECT: canonical host is `https://www.covercapy.com/`** but CLAUDE.md / config uses the apex
  `covercapy.com` (BASE_URL). Confirm `www` vs apex matches the canonical host actually served by
  Vercel; a mismatch (redirect from canonical host) splits signals. Pick one host everywhere.
- **DEFECT (duplicate content): the same practice exists at two URLs** ‚Äî
  `/treasure-island/maldonado-dentistry/` AND `/treasure-island/maldonado-dentistry-fl-cc3f24/`,
  each self-canonicalizing to itself. The hashed-suffix slug variant and the clean slug both shipped
  for the same office. This duplicates content site-wide wherever both slugs were generated.

## 8. CTA correctness

- Verify/insurance modal wired via `data-open="insurance"` (hero, rail, sticky, carrier rows) ‚Äî works.
  NOTE: the modal id/prefix here is `dlg-insurance` / `data-open`, NOT the `t5-verify-scrim`/
  `t5OpenVerify` IDs described in CLAUDE.md. Doc is stale OR template diverged; reconcile.
- "Official website" via `data-ext-site` button (exit handoff) ‚Äî present.
- Booking widget is a non-functional "Preview" until the office connects a scheduler (by design).
- Sticky bar secondary CTA -> `/compare-ppo-dental-plans` (correct). Edu link -> `/dental-insurance/`.
- Em-dashes appear in CTA-adjacent copy and FAQ answers ("Yes ‚Äî", "starting point ‚Äî") on BOTH
  templates ‚Äî CLAUDE.md violation.

## Prioritized defect list

### P0 (indexation / penalty risk)
- **P0-1 JS-only core content.** Carriers, treatments, hours, and nearby links are injected by JS
  from inline `T5`; static HTML ships empty containers. Server-render these into the HTML at build
  time. This is the single biggest SEO defect across all ~11,396 pages.
- **P0-2 Thin / near-duplicate body.** "About" prose is ~2 boilerplate sentences repeated corpus-wide.
  Risk of thin-content / doorway treatment at this scale.
- **P0-3 Duplicate URLs for one practice** (clean slug + `-st-xxxxxx` hashed slug both live, both
  self-canonical). De-duplicate: emit one URL per dentist and 301/canonical the other.
- **P0-4 Fabricated-rating exposure (verify).** Ensure `AggregateRating` is emitted ONLY when a real
  Google review count > 0; never default/synthesize a rating. Audit generator for the 0-review path.

### P1 (ranking / equity)
- **P1-1 No `/dentist` hub crumb or link.** Add "Find a Dentist" ‚Üí `/dentist` as breadcrumb
  position 2 (shift others down) and as a body/footer link, so 11k pages feed the national hub.
- **P1-2 No carrier-page links.** Convert listed carriers to anchors to /dental-insurance/{carrier}
  (or carrier silo pages) ‚Äî recover internal-link equity to carrier landing pages.
- **P1-3 No static sibling/nearby links.** Server-render the nearby-offices rail as real `<a>` links
  to neighboring dentist + city/metro pages for crawl discovery.
- **P1-4 Template fragmentation.** Two title/FAQ/section templates live (old "‚Äî Insurance Verified"
  em-dash vs new). Rebuild the whole corpus on one current template.

### P2 (polish)
- **P2-1 Em-dashes** in titles (old template) and body/FAQ copy (both) ‚Äî replace per CLAUDE.md.
- **P2-2 Title/description length.** Trim "Top-Rated ... in {City}, {ST}" titles and any meta > 155.
- **P2-3 H1 has no city/keyword.** Append city: e.g. `{Name} ‚Äî PPO Dentist in {City}, {ST}` (no
  em-dash; use comma) or keep H1 = name but add an H2 with the city+keyword early.
- **P2-4 Canonical host www vs apex** ‚Äî align to the served host.
- **P2-5 Redundant metro==city path segments** (`/ithaca/ithaca/`) ‚Äî cosmetic; consider collapsing.
- **P2-6 CLAUDE.md drift** ‚Äî modal IDs (`dlg-insurance`/`data-open`) and richer sections (booking,
  cost estimator, gallery, claim) are not reflected in the doc; update CLAUDE.md.

## Concrete fix patterns for generate-plans.js

1. **Server-render the data, don't only bake JS.** In `buildDentistPage`, emit static HTML for
   listed carriers (as links), services (as a `<ul>` of text), hours (a real table from `d.hours`),
   and the nearby rail (real `<a>` to sibling slugs). Keep `T5` JS only for modal/interactivity.
2. **Expand unique prose.** Generate 120‚Äì200 words of varied "About" copy weaving in the actual
   carrier names, services, doctor_name, rating, neighborhood, and weekend hours, with sentence
   templates randomized to reduce boilerplate similarity.
3. **De-dupe slugs.** One canonical URL per `dentists.id` (prefer clean `d.slug`); suppress or 301
   the hashed-suffix duplicate. Verify the sitemap lists only the canonical.
4. **AggregateRating guard:** `if (d.google_review_count > 0 && d.weighted_rating) { ...emit... }`
   else omit node entirely; never output a synthetic rating.
5. **Breadcrumb:** insert position-2 `{name:"Find a Dentist", item: BASE+"/dentist/"}` in both schema
   and visible nav; renumber subsequent positions.
6. **Carrier links:** map each listed carrier to its silo URL and render as `<a>` (both in the static
   carrier list and ideally in the About prose) to pass equity to carrier pages.
7. **One template, one host.** Rebuild full corpus on the current template; standardize canonical/OG
   host (apex vs www) to match Vercel's served host.
8. **Copy hygiene:** strip em-dashes from titles, descriptions, FAQ answers, and intros (replace with
   commas/colons). Enforce title <= ~60 and description <= ~155 chars via a truncation helper.
9. **H1:** include city/state keyword in H1 or an immediately-following H2.
