## WHAT YOU ARE BUILDING

You are upgrading the static SEO hub pages for CoverCapy — a luxury PPO dental concierge platform. The hub pages were initially generated by Claude Sonnet. Your job is to rebuild them with superior content quality, deeper editorial intelligence, and richer structured data, leveraging Opus 4.8's extended reasoning and long-context capabilities.

**The generator lives at:** `seo-build/generate-plans.js`  
**All hub functions to upgrade:** `buildMarketHub()`, `buildCityPage()`, `buildStateHub()`, `buildLocalAreaHub()`, `buildRegionalHub()`  
**Output:** Static `index.html` files inside `dental/{state}/{market}/{city}/`  
**Deploy:** `git push origin main` → Vercel auto-deploys

**Brand positioning:** "Get cover today, see a dentist tomorrow."  
**Feel:** Luxury concierge. Boutique hotel. NOT a healthcare portal.

- Missing `HowTo` schema for the verification flow
- No internal linking strategy beyond breadcrumbs
- No treatment-intent content (braces, implants, veneers, emergency)

```
/dental/{state}/                                    T3  — State hub

/dental/{state}/{market}/                           T4a — Metro hub (e.g. los-angeles)

/dental/{state}/{market}/{city}/                    T4c — City page
/dental/{state}/{market}/{city}/{dentist-slug}/     T5  — Dentist profile
```

**CRITICAL rules:**
- All trailing slashes. All served as `index.html` inside slug folder.
- Never use `d.seo_path` — it contains stale `/dentists/` prefixes. Build from parts: `stSlug + mkSlug + cSlug + dSlug`
- `d.slug` is canonical for dentist slugs (format: `practice-name-ca-xxxxxx`)
- Strip UTM from `d.website`: `.split('?')[0]`

### T3 — STATE HUB `buildStateHub()`

**Current:** Generic state description paragraph, market grid, plan rail.

**Opus improvements:**

**1. Extended reasoning for state-specific content strategy**
Use Opus's reasoning to generate non-generic, entity-rich state descriptions. Each state hub should:
- Reference specific metro population + dental access stats (where available from data)
- Name the 3 most dominant PPO carriers in that state's market
- Include a `serviceArea` schema with `GeoShape` or named cities
- Reference seasonal demand patterns (e.g. "snowbird season" for Arizona/Florida)
- Mention notable geographic coverage gaps as a trust signal ("rural Texas counties have limited PPO networks — CoverCapy focuses on verified urban markets")

**2. Richer FAQ generation**
Sonnet used 3 static FAQ templates. Opus should generate 5–7 FAQs per state, dynamically weighted by:
- Most common carriers in that state's dentist pool
- Whether the state has waiting-period-free plans available
- State-specific insurance law nuances (e.g. California has strong consumer protections)

**3. Internal linking intelligence**
State hubs should link to:
- Top 5 metros by office count (ranked, not alphabetical)
- Any T3.5 regional hubs for that state
- The 3 most search-intent-rich city pages (large population cities)
- The `/compare-ppo-dental-plans` page with state-specific anchor text

**4. Schema additions for state hubs:**
```json

  "about": {
    "@type": "MedicalCondition",
    "name": "Dental Care Access"
  },
  "audience": {
    "@type": "Patient"
  },
  "medicalAudience": "Patient",
  "specialty": "Dentistry"
}
```

### T4c — CITY PAGE `buildCityPage()`

**Current:** Dentist cards, city pill nav, plan rail, city FAQs (3 questions). Generic prose. No local personality.

**Opus improvements — this is the highest-leverage tier for SEO:**

**1. Hyper-local prose**
Each city page should feel written by someone who knows that city. Opus should use extended reasoning to generate city-specific signals:
- Nearest major hospital or medical center (proximity trust signal)
- Notable neighborhoods served ("Serving Fountain Valley, Garden Grove, and the Mile Square area")
- Dominant employer / insurance pool if identifiable (military bases, large employers, universities)
- HOA-heavy areas = high dual-coverage likelihood → mention FSA/HSA eligibility
- Retirement community proximity → mention dentures/implants/periodontics

This requires Opus to reason about geography, not just fill a template. This is exactly where Sonnet fell short — it produced identical-feeling city pages.

**2. Semantic entity clustering for Google NLP**
Each city page prose should co-occur these entity types:
- Dentist type entities: `general dentist`, `periodontist`, `oral surgeon`, `orthodontist`
- Procedure entities: `crown`, `implant`, `braces`, `whitening`, `root canal`, `cleaning`
- Insurance entities: `Delta Dental`, `UnitedHealthCare PPO`, `Guardian dental`, `Aetna`
- Location entities: city + nearby cities + zip codes + county name
- Urgency entities: `emergency dental`, `same-day appointment`, `walk-in dental`

Opus should place these naturally across H2s, prose, and FAQ answers — not as keyword stuffing but as genuine semantic coverage.

**3. Landmark-anchored intro sentence**
Pull from a city landmark lookup table (Opus should generate these during build for top 50 cities) and embed in the first sentence of the hero or about block. Examples:
- Fountain Valley: "...near Mile Square Park and the 405"
- Beverly Hills: "...steps from Rodeo Drive and Cedars-Sinai"
- Irvine: "...serving UC Irvine, the Spectrum, and the Irvine Company corridor"

**4. Upgrade FAQs to 5 questions.** Required additions:
- "What's the average cost of a dental cleaning in {city} without insurance?" — insert specific estimate + plan CTA
- "Which dentists in {city} accept {top_carrier}?" — dynamically pulls most common carrier from that city's dentist pool

**5. City-level `AggregateRating`**
Currently conditional on `total_reviews >= 10 && dentists.length >= 3`. Opus should lower the floor to `total_reviews >= 5` and include a `reviewCount` note in the prose.

**6. Schema addition: `LocalBusiness` aggregate for the city**
```json
{
  "@type": "Dentist",
  "name": "PPO Dentists in {city}",
  "areaServed": { "@type": "City", "name": "{city}", "containedInPlace": { "@type": "State", "name": "{state}" } },
  "hasMap": "https://www.google.com/maps/search/ppo+dentist+{city}+{state}"
}
```

## HOW TO LEVERAGE OPUS 4.8 SPECIFICALLY

Opus 4.8 has capabilities Sonnet 4 lacks. Use them explicitly:

### 1. Extended reasoning for content generation
Before generating city prose, Opus should think through:
- What is the dominant demographic in this city? (retirement, young families, military, university)
- Which 3 PPO carriers appear most in the dentist pool for this city?
- What is the most likely life event driving dental search here? (new job, baby, layoff, emergency)
- Which 2–3 procedures dominate this city's dentist profiles?
- What nearby landmarks or employers create geo-authority signals?

This reasoning step produces prose that doesn't feel templated. This is the core competitive advantage over Zocdoc and DentalPlans.com — they are databases. CoverCapy should feel editorial.

### 2. Long-context data synthesis for FAQs
Opus can hold the full dentist dataset for a market in context. Use this to generate data-grounded FAQs rather than templates:
- "14 of the 23 PPO dentists in Fountain Valley accept Delta Dental PPO" — pulled from real records
- "The highest-rated office in {city} has {X} reviews averaging {Y} stars" — real data citation
- "Weekend appointments are available at {N} of {total} offices in {city}" — from `open_weekends`

Sonnet used fixed FAQ templates. Opus must generate from data.

### 3. Semantic coherence across tiers
Opus should ensure that:
- A T4c city page does not contradict its parent T4a metro hub in office counts or carrier claims
- FAQs at T4a don't repeat verbatim at T4c (different specificity levels)
- State-level claims are consistent with market-level claims

Sonnet had no cross-tier coherence. Opus should run a consistency pass before finalizing.

### 4. Structured output validation
For each hub function, Opus should validate JSON-LD structure before embedding. Use internal tags during generation:
```
<schema_check>Does every @type have a required property? Are all @id values unique per page? Is sameAs an array?</schema_check>
```
Sonnet occasionally produced `sameAs` as a string instead of array. Opus must self-check.

### 5. Iterative improvement loop
After generating a city page, Opus should ask:
- Does the first paragraph of prose contain at least 2 real location signals?
- Are there at least 5 FAQ entries?
- Does the page differ meaningfully from the metro hub page above it?
- Are there any em-dashes in the copy?
If any check fails, revise before outputting.

## SUCCESS CRITERIA

A hub page built by Opus 4.8 should:
- Pass a "could this have been written by a local?" test for city prose
- Have zero verbatim sentence repeats across 3 city pages in the same metro
- Reference at least 2 real data points from the dentist pool (carrier counts, review aggregates)
- Include 5+ FAQs, at least 2 data-grounded
- Validate with Google Rich Results Test (schema must pass)
- Feel luxury — editorial, concierge, never healthcare portal
- Score no AI-writing flags on standard AI detection tools (vary sentence length, use specifics, avoid hedging language)
